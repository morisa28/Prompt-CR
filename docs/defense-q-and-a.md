# 答辩问答准备

## 1. 你们和普通 Prompt 模板网站有什么区别？

普通模板网站主要提供固定格式。本项目先评分诊断，再动态追问，随后生成 Prompt 并自审，最后把低分模式写入错题本，形成“需求澄清 + 评分 + 生成 + 自审 + 迭代建议”的闭环。

## 2. 你们和 Codex、Cursor、Copilot 有什么区别？

Codex、Cursor、Copilot 是执行或辅助写代码的工具。Prompt-CR 不替代它们，而是帮助用户把模糊需求整理成适合这些 coding agent 执行的工程化任务 Prompt。

## 3. 为什么评分维度是这 8 个？

这 8 个维度对应 coding agent 成功执行任务所需的关键条件：目标、上下文、可定位输入、约束、输出格式、验收标准、风险处理和工具适配。它们来自软件工程任务描述中的常见失败点。

## 4. 评分是否可靠？如何验证？

当前评分器是规则型 MVP，已加入反作弊和具体证据识别。后续会用 30 条 eval cases、人工评分一致性实验和 coding agent 执行效果实验校准可靠性。

## 5. 错题本是否真的能让系统进步？

当前错题本不会自动学习或自动改规则。它会统计高频低分维度和错误类型，并生成问题树、评分规则、模板的修改建议，作为半自动迭代机制。

## 6. 为什么只聚焦 3 个核心场景？

Bug 修复、新功能开发和测试生成覆盖“发现问题、实现功能、验证质量”的软件工程闭环，频率高、可展示、可评估，适合作为大创 MVP。

## 7. 如果用户乱填，系统如何处理？

评分器会识别关键词堆砌、空泛占位符和“不要乱改”“测试通过”等低质量表达，并给出 warning 和低分反馈。

## 8. 如果用户没有日志、文件路径和测试命令怎么办？

系统会优先追问这些信息。如果用户确实没有，生成的 Prompt 会要求 agent 先复现、收集日志或从项目配置识别测试命令，并在无法验证时诚实报告原因。

## 9. 项目的创新性在哪里？

创新点不在“生成一个 Prompt”，而在把 Prompt 写作转化为需求工程训练，并通过评分、追问、自审和错题本形成可复盘闭环。

## 10. 结题时如何证明项目成功？

用三类证据：可运行系统演示、30 条以上 eval cases 和评分报告、用户或人工评分实验数据。重点观察评分提升、缺失字段下降和 agent 执行质量变化。
