# Prompt Rewrite Examples

## Example 1: 原始需求到 Codex 强执行 Prompt

用户原始需求：

```text
帮我优化这个 prompt，让 Codex 更好执行。
```

输出：

```text
你是 Codex，请把我提供的原始 prompt 改写为可直接执行的 Codex prompt。

输入材料：
- 原始 prompt：[在这里粘贴]
- 目标任务类型：[代码修复/功能开发/重构/文档分析/PDF 分析/计划模式/其它]
- 目标工作目录：[path，如未知写“待补充”]

执行步骤：
1. 诊断原 prompt 是否缺少任务目标、上下文、输入材料、执行顺序、约束、输出格式、验收标准和自检要求。
2. 将模糊词替换为具体动作和可检查结果。
3. 对缺失但会影响执行的信息，用 `[待补充: ...]` 标记。
4. 如果缺失信息会导致误改代码、删除数据或错误使用工具，先列为阻塞问题。
5. 输出重写后的 Codex prompt。

约束：
- 必须保留原始意图。
- 禁止添加原 prompt 没有依据的新功能。
- 禁止把不确定事实写成确定条件。
- 如果任务是 plan 模式，必须要求先分析不改文件，并等待用户确认。

输出格式：
1. 原 prompt 的主要问题：用表格列出问题、风险、修复方式。
2. 重写后的 prompt：放入一个 `text` 代码块。
3. 需要用户补充的信息：只列会影响执行的内容。

验收标准：
- 重写后的 prompt 包含工作目录、任务目标、输入材料、执行步骤、约束、输出格式、验收标准和自检要求。
- Prompt 能直接交给 Codex 执行。
- 所有不确定信息已标注。
```

## Example 2: Vue 项目报错修复 Prompt

用户原始需求：

```text
我的 Vue 项目报错了，帮我修。
```

输出：

```text
你是 Codex，请修复这个 Vue 项目中的报错。

工作目录：
[Vue 项目路径]

报错信息：
[粘贴完整终端报错、浏览器控制台报错或构建失败日志]

复现步骤：
1. 运行：[npm run dev / npm run build / npm test]
2. 打开页面或执行操作：[route/action]
3. 当前结果：[actual]
4. 期望结果：[expected]

相关文件：
- package.json
- vite.config.js
- src/main.[js|ts]
- src/App.vue
- [报错栈中出现的组件或路由文件]

执行步骤：
1. 读取 package.json、Vite 配置、入口文件和报错栈指向的 Vue 组件。
2. 复现或静态定位报错根因，区分是语法、类型、依赖、路径、组件状态、生命周期还是构建配置问题。
3. 给出最小修复方案。
4. 修改必要文件，禁止无关重构、全局格式化或删除现有功能。
5. 运行验证命令：[npm run build]，如果有测试则运行：[npm test]。
6. 如果验证失败，继续修复由本次改动引入的问题；如果失败来自环境或既有问题，报告证据。

约束：
- 必须保持现有 UI 路由和用户交互不变，除非报错根因要求改变。
- 禁止新增依赖，除非说明无法用现有依赖修复并等待用户确认。
- 禁止删除测试或降低断言。
- 必须保护用户已有改动，不得 revert 未授权修改。

输出格式：
- 根因
- 修改文件
- 关键改动
- 验证命令与结果
- 剩余风险或需要用户确认的问题

验收标准：
- 报错不再出现。
- 指定页面或命令可正常运行。
- 构建或测试通过，或无法运行原因明确。
- 修改范围与根因匹配。
```

## Example 3: PDF 方法提炼为 Skill Prompt

用户原始需求：

```text
读这个 PDF，总结里面的技巧。
```

输出：

```text
你是 Codex，请深度阅读指定 PDF，并把其中的技巧工程化为一个可复用的 Codex skill。不要只做章节摘要。

PDF 路径：
[pdf_path]

目标 skill 路径：
[skill_path]

核心目标：
将 PDF 中的方法、规则、案例、反例、注意事项和迭代思路转化为 Codex 后续可调用的规则、流程、模板、检查表和示例。

执行顺序：
1. 扫描当前目录中的 PDF，确认目标 PDF 是否存在，并说明选择理由。
2. 检查是否已有相关 skill；如存在，优先扩展；如不存在，创建不冲突的新 skill 目录。
3. 第一轮结构阅读：提取目录、章节、核心主题、重点章节和低优先级章节。
4. 第二轮技巧提取：提取任务目标、上下文、输入、角色、约束、输出格式、示例、反例、迭代和自检技巧。
5. 第三轮规则转化：把技巧转为 Codex 行为规则、工作流、验收标准和风险控制。
6. 第四轮 skill 适配：删除产品无关细节，保留能指导 Codex 执行复杂任务的规则。
7. 创建或更新 SKILL.md；内容较长时拆分 templates.md、checklists.md、examples.md，并在 SKILL.md 中链接。
8. 删除重复、空泛、无法执行的内容。
9. 用三个调用场景模拟验证；发现缺口后修改 skill。

硬性约束：
- 禁止直接大段复制 PDF 原文。
- 禁止只写读书笔记。
- 禁止在未扫描目录和已有 skill 前写文件。
- 禁止把 PDF 没有支持的内容写成确定结论。
- 必须包含使用场景、不使用场景、核心原则、构造流程、优化流程、plan 模式规则、复杂任务拆解、上下文收集、约束写法、输出格式、验收标准、自检、常见问题修复、模板、Codex 规则、示例和最终检查表。

输出格式：
- 分析对象
- 创建或修改的文件
- 核心规则
- 模板数量
- 示例数量
- 模拟测试结果
- 不确定或需人工确认的问题

验收标准：
- Skill 可长期复用。
- 模板能直接复制使用。
- 检查表能判断 prompt 是否合格。
- 示例覆盖 prompt 重写、代码修复、PDF 转 skill。
- Plan 模式不会被写成立即执行模式。
```

## Example 4: 3D 旋钮拖拽功能 Prompt

用户原始需求：

```text
帮我做一个 3D 交互网页里的旋钮拖拽功能。
```

输出：

```text
你是 Codex，请为现有 3D 交互网页实现旋钮拖拽功能。

工作目录：
[project_path]

目标：
让用户在网页中拖拽指定 3D 旋钮，并将拖拽结果映射为数值变化或状态变化；必须保留现有场景、资源和其它交互。

对象与文件：
- 旋钮对象名称：[object_name]
- 3D 场景或 Spline/Three.js 资源：[asset_or_scene_path]
- 相关组件：[component_paths]
- 状态或业务逻辑文件：[state_or_logic_paths]

当前行为：
- 当前异常或缺口：[none/current_issue]
- 现有可用交互：[list]

目标交互：
- 触发方式：鼠标或触摸拖拽旋钮。
- 映射规则：顺时针/向右拖拽增加值，逆时针/向左拖拽减少值。
- 数值范围：[min] 到 [max]。
- 边界行为：到达最小或最大值后停止继续变化。
- UI 反馈：[label/value/animation/highlight]

视觉与验证要求：
- 初始视口：[1440x900 and mobile size if required]
- 初始视角：[camera/view]
- 旋钮必须可见且不被 UI 遮挡。
- 如允许浏览器自动化，请启动 dev server 并截图验证；如不允许，输出用户手动截图验证步骤。

执行步骤：
1. 读取 3D 场景加载、组件状态、事件绑定和样式文件。
2. 确认现有交互和资源引用，避免破坏其它对象。
3. 设计最小实现方案，包括 pointer 事件、拖拽状态、数值映射和清理逻辑。
4. 实现旋钮拖拽。
5. 验证桌面和移动交互；检查拖拽开始、拖拽中、释放、越界和重复拖拽。
6. 输出修改文件、验证结果和残留风险。

约束：
- 禁止替换整个 3D 场景。
- 禁止删除现有对象、动画、路由或资源。
- 禁止引入新 3D 库，除非现有技术栈无法实现并获得用户确认。
- 必须保持文本、按钮和 3D 对象不重叠。

输出格式：
- 实现方案
- 修改文件
- 交互映射
- 验证结果
- 用户手动验证步骤
- 剩余风险

验收标准：
- 拖拽旋钮能稳定改变目标值。
- 值不会越过边界。
- 释放鼠标或触摸后不会继续拖动。
- 现有场景和交互保持可用。
- 视觉检查确认对象可见、页面非空、UI 不重叠。
```
