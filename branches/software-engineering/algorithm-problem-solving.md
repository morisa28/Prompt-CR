# Algorithm Problem Solving

## 1. Purpose

用于把算法与数据结构题目转化为可交给 AI 工具分析、编码和验证的 prompt。重点是澄清输入输出、数据范围、复杂度要求、边界用例、证明思路和测试方式。

## 2. Trigger Conditions

- 用户要求解决算法题、数据结构题、竞赛题或面试题。
- 用户要求分析时间复杂度、空间复杂度、边界条件或证明正确性。
- 用户希望生成可运行代码和测试用例。

## 3. Required Inputs

- `{{problem_statement}}` 题目描述。
- `{{input_format}}` 输入格式。
- `{{output_format}}` 输出格式。
- `{{constraints}}` 数据范围和复杂度要求。
- `{{examples}}` 样例输入输出。
- `{{target_language}}` 目标语言。
- `{{verification_cases}}` 验证用例。

缺失信息处理：
- 数据范围未知时，必须要求补充或列出复杂度假设。
- 样例缺失时，先构造最小样例、边界样例和异常样例，但标注为假设。
- 目标语言未知时，默认先输出思路与伪代码。

## 4. Prompt Construction Rules

- 先复述题意，区分输入、输出、目标和限制。
- 必须给出至少一种可行算法，并说明为什么满足复杂度。
- 必须列出边界条件，例如空输入、单元素、重复值、极值和非法输入。
- 代码实现必须匹配目标语言和输入输出格式。
- 验证必须包含样例、边界用例和复杂度压力用例。

## 5. Hard Constraints

- 禁止忽略数据范围直接给暴力解。
- 禁止只给代码不解释复杂度。
- 禁止把未经验证的复杂度写成确定结论。
- 禁止省略边界用例。
- 必须包含可检查的验收标准。

## 6. Output Format

最终 prompt 应要求目标模型输出：

```text
题意复述：
关键约束：
算法思路：
正确性说明：
复杂度分析：
代码实现：
测试用例：
未覆盖风险：
```

## 7. Quality Checklist

- [ ] 输入输出格式明确。
- [ ] 数据范围明确或被标注为待补充。
- [ ] 算法复杂度满足约束。
- [ ] 边界用例覆盖完整。
- [ ] 代码与目标语言和 I/O 格式一致。
- [ ] 验证用例可运行。

## 8. Common Mistakes

| Common Mistake | Risk | Repair |
| --- | --- | --- |
| 不问数据范围 | 算法可能超时 | 把 constraints 作为必填字段 |
| 只有代码无解释 | 无法教学和复查 | 输出思路、正确性和复杂度 |
| 缺边界用例 | 隐藏错误 | 加空、极值、重复和最小规模用例 |

## 9. Reusable Template

```text
你是 {{target_ai_tool}}，请解决以下算法任务。

题目：{{problem_statement}}
输入格式：{{input_format}}
输出格式：{{output_format}}
数据范围：{{constraints}}
样例：{{examples}}
目标语言：{{target_language}}

请先澄清题意和约束，再给出算法思路、正确性说明、复杂度分析、代码实现和测试用例。若数据范围缺失，先标注 `[待补充: constraints]`，不要假设暴力解一定可行。
```

## 10. Example

用户需求：

```text
把两数之和写成 Python 代码，并解释复杂度。
```

生成策略：补充输入输出、数据范围、重复元素、无解处理和测试用例，再要求输出哈希表解法、复杂度和 Python 实现。
